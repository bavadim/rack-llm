# Зафиксировать публичный API и границы библиотеки

Status: done

## SMART goal

За 1 рабочий день описать и зафиксировать публичный API v1 так, чтобы все дальнейшие задачи реализовывали один и тот же минимальный интерфейс без экспериментальных модулей.

## Dependencies

Нет жестких зависимостей, кроме сохранения архитектурных принципов из `ARCHITECTURE.md`.

## Scope

- Создать/обновить `docs/api.md`.
- Описать публичные функции: `lit`, `rx`, `seq`, `select`, `repeat`, `text`, `rank`, `ban`, `weight`, `pure`, `bind`, `generate`, `generate-stream`, `check`, provider constructors.
- Описать внутреннюю семантику `Guide[A]`, `Watch`, `Ranked`, `Banned`.
- Явно написать, что `text` — единственное место open-world.
- Явно написать, что `rank` не открывает мир.

## Out of scope

- Не реализовывать код.
- Не писать IFBench/экспериментальные runner'ы.
- Не добавлять новые DSL-операторы сверх API v1.

## Public interfaces / touched interfaces

Документ должен описывать сигнатуры в стиле:

```racket
(lit s) -> Guide[Void]
(rx pattern) -> Guide[String]
(text watch ...) -> Guide[String]
(rank w expr) -> Ranked
(ban expr) -> Banned/Watch
(generate provider prompt guide ...) -> GenerationResult
```

## Scientific / design notes

Научная суть: hard guidance и soft guidance должны быть частными случаями одного log-score подхода. Hard-правило дает `0/-inf`, soft-наблюдатель дает конечный score. Это заменяет старую схему произвольных Python-эвристик на структурные, анализируемые правила.

## Implementation notes

- Сохранять функциональный минимализм: структуры + чистые функции, без объектных иерархий.
- Не добавлять экспериментальные зависимости.
- Ошибки должны быть явными и локальными: лучше понятный `raise-argument-error`, чем молчаливое “примерно работает”.
- Все новые публичные имена должны быть отражены в документации API.
